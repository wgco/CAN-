STM32 + CAN 分布式机器人控制架构技术文档

Bus RTOS / CANopen-like / CiA402 / PDO实时调度工程设计

1. 文档目标

本文档旨在阐述如何从基本的 STM32 CAN 通信底层出发，升级构建一套面向高动态机器人（如四足步态机器人、多自由度协作臂）的分布式总线控制系统。核心目标不是实现简单的“能发 CAN 报文”，而是实现具备总线负载可观测性、抗拥塞调度和实时控制数据确定性的 Bus RTOS 实时总线调度系统。

2. 核心认知：总线即共享实时资源

在机器人级控制总线设计中，必须树立核心认知：CAN 绝不是一个简单的通信接口，而是一种稀缺的、共享的实时物理资源。

等价交换律：CPU 调度算力本质是分配“时间片”；CAN 总线调度带宽本质是分配“Bit 时间”。

任何一个分布式节点发送非必要报文，都在掠夺和占用总线的绝对时间资源。因此，总线上的流量必须同 CPU 线程一样，接受严格的调度（Scheduling）、限流（Rate Limiting）、监控（Monitoring）与降级（Degradation），否则物理总线必然陷入不可控状态。

3. 基础架构拓扑

系统采用数据流向与调度边界解耦的四层软件拓扑架构：

应用层 (步态规划 / 轨迹插补 / 核心状态机) │ ▼ Command Aggregator (命令汇聚器) │ ▼ CAN Scheduler (总线调度器内核) │ ┌──────────────────────┴──────────────────────┐ ▼ ▼ ┌──────────────────┐ ┌──────────────────┐ │ PDO Buffer │ │ FIFO Queues │ │ (实时覆盖型缓冲区)│ │ (突发/事务型消息) │ └──────────────────┘ └──────────────────┘ │ │ └──────────────────────┬──────────────────────┘ ▼ bxCAN / FDCAN 硬件发送邮箱 (Mailboxes) │ ▼ CAN BUS 物理总线 (1 Mbps 差分网络)

4. 为什么不能采用单一发送 FIFO Queue

初学者和低端系统常将所有待发数据（控制帧、诊断帧、心跳、配置）统一压入一个软件 FIFO 队列。在总线通畅时该方案能正常工作，但在多轴联动或电磁干扰下，该设计会导致控制延迟的灾难性累积。

假设控制环周期为 1ms，主控连续下发物理 ID 为 100、110、120、130 的控制指令。一旦总线发生瞬时物理仲裁碰撞或突发单帧错误重发，这些消息在单一 FIFO 内便会形成无差别积压：

[ 软件单一FIFO队列积压流 ] 进入队尾: 100(t=0ms) ──> 110(t=1ms) ──> 120(t=2ms) ──> 130(t=3ms) 总线恢复后依次弹出: 发送 100(旧) ──> 发送 110(旧) ──> 发送 120(旧) ──> 发送 130(最新)

物理总线恢复正常后，系统继续顽固地发送已失去时效性的历史控制指令，导致最新的控制命令（130）排在队尾被迫等待。这种控制延迟的阶跃累积反应在机器人肢体上，表现为关节剧烈抖动、电机高频抽动、闭环稳定性（Phase Margin）严重降级。

5. PDO 与 FIFO 的工程解耦关系

本架构中，实时过程数据（PDO）与异步事务事务数据（FIFO）并非替代关系，而是共存与协同关系。实时命令不入排队 FIFO，而是采用覆盖型 Buffer；事务消息则利用不同级别的 FIFO 进行缓冲。

特性维度 实时过程数据缓冲 (PDO Buffer) 事务/异步消息队列 (FIFO Queue)
数据承载关系 严格的覆盖关系 (Overwrite) 严格的累加关系 (Append)
历史数据保留 否。新控制量直接覆盖并抹除历史未发值 是。队列中的每一帧事务消息均需被保留送达
底层写入行为 新数据直接原地刷新物理内存空间 新数据写入队尾，旧数据留存在队头等待弹出
典型应用场景 电机目标位置、目标转矩、实际传感器反馈 SDO配置、Event突发告警、Heartbeat保活帧

6. PDO 核心原则：实时且不可排队

“控制帧绝不排队”是本系统的核心铁律。控制层只确保总线发送的是当前系统计算出的最新切片值。以下是该机制的纯 C 语言工程无锁/半原子操作原型实现：

#include <stdint.h>
#include <string.h>

/**
 * @brief 实时过程数据(PDO)覆盖型缓冲区结构体
 */
typedef struct {
    uint32_t cob_id;          // CAN 对象的通信对象标识符
    uint8_t  data[8];         // 原始 Payload 数据空间
    uint8_t  dlc;             // 数据长度
    volatile uint8_t updated; // 原子更新标志位
} PDO_Buffer_t;

static PDO_Buffer_t s_tx_pdo_buffer[16]; // 映射多轴电机的 PDO 缓冲表

/**
 * @brief 应用层高频计算线程：写入最新控制指令（全覆盖）
 */
void BusRTOS_Write_PDO(uint8_t node_index, uint32_t cob_id, const uint8_t* p_cmd, uint8_t dlc) {
    s_tx_pdo_buffer[node_index].cob_id = cob_id;
    // 覆盖写入最新的内存数据，历史未发出的数据在此处被合法抛弃
    memcpy((void*)s_tx_pdo_buffer[node_index].data, (const void*)p_cmd, dlc);
    s_tx_pdo_buffer[node_index].dlc = dlc;
    s_tx_pdo_buffer[node_index].updated = 1; // 置位更新标志
}

7. 优先级倒置与饥饿问题 (Priority Starvation)

如果调度器无脑优先发送 PDO 消息（例如：if (has_pdo) send_pdo(); else if (has_sdo) send_sdo();），由于机器人在作业时 PDO 往往以 1ms 的频率持续占满通道，这会导致总线发生优先级饥饿（Priority Starvation）。处于更低优先级的异步事务（如状态监控 Heartbeat、在线参数调整 SDO、Event 突发异常告警）将无法获得总线控制权而被完全“饿死”，导致系统监控层发生伪断线报警。

8. 引入总线时间预算机制 (Bus Budget)

为了解决饥饿问题，系统引入总线预算机制（Bus Budget Allocation）。在 1 Mbps 的标准工业波特率下，底层物理帧的理论时间消耗可通过如下数学公式精确推导：

CAN标准帧物理时间计算：
$T_{\text{frame}} = \frac{34 + 8 \times \text{DLC} + \text{StuffingBits}}{1,000,000\text{ bps}}$
一帧标准数据帧（DLC = 8）包含位填充后大约消耗 130 bits。在 1ms 的控制周期内，总线最大可容纳物理极限为：$1000\,\mu\text{s} / 130\,\mu\text{s} \approx \mathbf{7.69}$ 帧报文。

为确保总线利用率维持在安全线以下，每个 1ms 时间预算周期分配如下：

typedef struct {
    uint8_t pdo_budget;   // 过程数据帧预算（控制指令）
    uint8_t event_budget; // 突发告警帧预算
    uint8_t sdo_budget;   // 配置事务帧预算
    uint8_t hb_budget;    // 心跳保活帧预算
} Bus_Budget_t;

// 1ms 周期内的硬性安全时间预算表分配：共 7 帧
static const Bus_Budget_t c_safe_budget = {
    .pdo_budget   = 4,    // 4 * 130us = 520us (主控制带宽)
    .event_budget = 1,    // 1 * 130us = 130us
    .sdo_budget   = 1,    // 1 * 130us = 130us
    .hb_budget    = 1     // 1 * 130us = 130us （总占用时间 910us < 1000us）
};

9. 总线调度器（CAN Scheduler）核心实现

以下是具备时间预算限制和限流保护的 CAN_Scheduler 核心调度引擎源码：

void BusRTOS_CAN_Scheduler(void) {
    static Bus_Budget_t active_budget;
    static uint32_t last_tick = 0;
    
    uint32_t current_tick = Get_System_Tick_MS();
    if (current_tick != last_tick) {
        // 核心：每到新的 1ms 边界，强行重置刷新发送配额，重振总线生命力
        active_budget = c_safe_budget;
        last_tick = current_tick;
    }

    /* 1. 优先调度实时过程数据 (PDO) */
    if (active_budget.pdo_budget > 0) {
        for (int i = 0; i < 16; i++) {
            if (s_tx_pdo_buffer[i].updated) {
                if (HAL_CAN_Transmit_To_Hardware(&s_tx_pdo_buffer[i])) {
                    s_tx_pdo_buffer[i].updated = 0;
                    active_budget.pdo_budget--;
                    return; // 轮询投递单帧后退出，保护硬件邮箱不发生突发溢出
                }
            }
        }
    }

    /* 2. 在配额限制下，异步调度排队型事务 FIFO */
    if (active_budget.event_budget > 0 && !Is_FIFO_Empty(&g_event_fifo)) {
        CAN_MSG_t msg = Pop_From_FIFO(&g_event_fifo);
        if (HAL_CAN_Transmit_Raw(&msg)) {
            active_budget.event_budget--;
            return;
        }
    }

    if (active_budget.sdo_budget > 0 && !Is_FIFO_Empty(&g_sdo_fifo)) {
        CAN_MSG_t msg = Pop_From_FIFO(&g_sdo_fifo);
        if (HAL_CAN_Transmit_Raw(&msg)) {
            active_budget.sdo_budget--;
            return;
        }
    }

    if (active_budget.hb_budget > 0 && !Is_FIFO_Empty(&g_hb_fifo)) {
        CAN_MSG_t msg = Pop_From_FIFO(&g_hb_fifo);
        if (HAL_CAN_Transmit_Raw(&msg)) {
            active_budget.hb_budget--;
            return;
        }
    }
}

10. 为什么需要总线负载可观测性

传统的 STM32 bxCAN 物理控制器只会向应用层上报 `TXOK`（发送完成）或硬件报错（如 `Bus-Off`、`Passive Error`）。它完全是盲目的，不会主动提供当前物理网络的带宽拥堵率和实时压力情况。因此，架构层必须通过软件算法显式计算出实时总线负载（Real-time Bus Load）。

11. 负载（Bus Load）的工业级工程定义

在实时工业通信体系中，总线负载的科学定义绝对不是“当前发送帧数占总允许帧数的比例”。因为网络中可能穿插混杂着 DLC 从 0 到 8 不等长度的复杂报文，短帧与长帧占用物理铜线的时间大不相同。

正统定义：总线负载 $\equiv$ 物理总线被显性差分电平占用的绝对时间 / 统计总窗口时间。

12. 基于 Bit 时间占用率的数学推导

以 1 Mbps 波特率为例，1 bit 的物理传输持续时间正好等于 $1\,\mu\text{s}$。一帧携带 8 字节数据的标准 CAN 帧，其包含了 11 位标识符、控制位、CRC校验、ACK段以及极端情况下的位填充（Bit Stuffing），经验物理总长度约为 130 bits，即占用总线时间 $130\,\mu\text{s}$。

如果在 1ms 的观察窗口内，总线成功发送或监听到了 5 帧完整长帧，则累计占用总线时间为：$5 \times 130\,\mu\text{s} = 650\,\mu\text{s}$。当前周期的负载率为：

Bus Load = 650 μs / 1000 μs = 65.0%

13. STM32 物理层负载统计监控实现

为消除频繁浮点运算，系统在硬件层采用滑动窗口积分累加法（Window Accumulation）实现负载统计：

typedef struct {
    volatile uint32_t used_bits_accumulator; // 窗口内消耗的物理 bit 累加器
    volatile uint8_t  current_bus_load_pct;  // 转换后的百分比负载率 (0 - 100%)
} Bus_Monitor_t;

static Bus_Monitor_t g_bus_monitor = {0};

/**
 * @brief 核心要素8 & 要素9：底层 bxCAN / FDCAN 发送完成及接收中断的轻量级监控打标
 * @note 必须在每次硬件中断安全写入，单次开销 < 5 个 CPU 周期
 */
inline void BusRTOS_Log_Frame_To_Monitor(uint8_t dlc) {
    // 快速近似物理特征方程：时间开销转化公式 bits = 34 + (8 * dlc) + 估计填充位
    uint32_t bits_spent = 34 + (dlc << 3) + 12; 
    g_bus_monitor.used_bits_accumulator += bits_spent;
}

/**
 * @brief 核心要素7：在 10ms 滑动窗口定时器中执行无浮点负载解算
 */
void BusRTOS_Calc_BusLoad_10MS_Timer(void) {
    // 10ms 窗口在 1Mbps 下的最大理论带宽为 10,000 bits
    uint32_t total_capacity_bits = 10000;
    
    uint32_t spent = g_bus_monitor.used_bits_accumulator;
    g_bus_monitor.used_bits_accumulator = 0; // 快速清空进入下一滑动统计窗口
    
    // 防止越界异常保护
    if (spent > total_capacity_bits) spent = total_capacity_bits;
    
    // 纯整数运算：直接规避昂贵的浮点计算单元(FPU)开销
    g_bus_monitor.current_bus_load_pct = (uint8_t)((spent * 100) / total_capacity_bits);
}

14. 为什么物理总线会发生突发性拥塞（Burst Congestion）

总线突发性拥塞通常由以下运行状态衍生：

  1. 多节点同时上电（Bootup Burst）：所有外设电机上电瞬间同时向总线投递初始化就绪报文。
  2. 网络错误重发（ACK Failure/Error Passive）：物理层由于强电磁干扰、接触不良导致 ACK 错误，bxCAN 发送器触发硬件无限自动重发（Automatic Retransmission）。
  3. 非实时事务重叠（SDO Congestion）：主控高频调试读写参数，多个进程同时压入 SDO，导致总线利用率瞬间突破 95% 临界线。

15. 工业级抗拥塞系统设计

针对突发拥塞，本规范制定了四个闭环层级的系统保障机制：

  • 相位错开（Phase Shifting）：通过微秒级延时，错开各电机的物理投递节拍，从源头消灭仲裁碰撞。
  • 总线限流（Rate Limiting）：事务消息在入队前施加严格的发送周期频率限制。
  • 预算管控（Budget Gatekeeping）：利用前述的 1ms 发送配额机制封锁多余流量。
  • 动态降级（Dynamic Degradation）：当观测到实时总线负载率突破 85% 警戒线时，系统自动切断低优先级 SDO 与诊断帧的发送。

16. 核心控噪技术：相位交错（Phase Shift）

绝不允许主控在定时器 0us 时刻无脑给全网 12 个电机同时投递控制指令。这会导致 12 帧报文在物理层疯狂碰撞进行硬仲裁，抬高总线瞬时噪声。应通过时间轴交错，实施微秒级相位错开（Phase Shift Offsetting）：

[ 错误高强碰撞模式 (0us同时投递) ] 0us: ──> [Motor1, Motor2, Motor3, Motor4, Motor5, Motor6] ──> 发生强仲裁碰撞 [ 规范相位交错模式 (100us时基细分) ] 0us : ──> Motor1 发送 100us: ──> Motor2 发送 200us: ──> Motor3 发送 300us: ──> Motor4 发送 (总线差分波形平滑分布，物理层噪声降至最低)

分布式控制中，主控不再负责复杂的 PWM 桥路开关或闭环 FOC 矢量计算，而是专注于高层规划。

18. 分布式控制的工业本质

分布式控制的本质是：主控下发目标期望量（Desired Trajectory Slice），节点在本地执行超高频闭环算法。主控严禁插手单轴电机的细微瞬态控制。主控下发的只能是“在 $t = 34\,\text{ms}$ 时关节 1 必须到达位置 $\theta_1$”，电机节点内部依赖其 20 kHz 的电流环独立对抗现场反向负载扰动。

19. PDO 与 SDO 的工程职责划分

  • PDO (Process Data Object，过程数据对象)：用于高频、无单帧确认、单向发布的控制指令与底层反馈。例如目标位置、实际力矩、编码器多圈计数值。
  • SDO (Service Data Object，服务数据对象)：用于低频、带物理硬确认、点对点读写的配置项。例如修改电机减速比、设置过流保护门限、在线捞取错误历史日志。
  • Heartbeat (节点守护心跳)：分布式电机定期对外广播其当前 CiA402 状态机状态，用于主控网络层评估节点拓扑的物理健康度。

20. 机器人总线控制系统：三条工程铁律

第一铁律：控制帧绝不排队 —— 宁可丢失旧控制块，也绝不发送积压的历史脏指令。
第二铁律：总线负载必须可观测 —— 通信网络不能是黑盒，每一比特的空间消耗必须置于软件监控之下。
第三铁律：非实时消息必须可降级 —— 当带宽面临灾难冲突时，事务消息必须为主控制流绝对让步。

21. 最终总结：系统本质是 Bus RTOS

综上所述，高动态多轴机器人的底层开发，其本质根本不是编写一个“CAN 通信收发程序”，而是构建一个 Bus RTOS（实时总线调度操作系统）。它通过在物理层和应用层之间建立标准的对象字典映射、无锁环形缓冲区、时间预算拦截器以及相位交错引擎，使得脆弱、不确定性的串行网络转化为机器人级高确定性、硬实时性、具备确定 Phase Margin 的共享计算资源网络。
