核心认知：CAN 是共享实时资源

CAN 总线不是普通通信接口。从资源调度角度看：

类比维度CPU / RTOSCAN 总线
共享资源CPU 时间片总线 bit 时间
任务单元线程 / 任务CAN 帧
调度器OS SchedulerCAN_TxScheduler
资源计量CPU 占用率Bus Load %
过载处理任务降级 / 抢占帧降级 / Budget 限流
本质结论 系统不是"CAN通信程序"，而是 Bus RTOS——实时总线调度系统。必须像管理CPU资源一样管理总线资源：调度、限流、监控、降级。

typedef enum {

    CAN_BUS_2 = 1,
    CAN_BUS_COUNT
} CanBusId;

typedef enum {
    CAN_MSG_PDO,        // 实时控制帧，覆写槽语义
    CAN_MSG_EVENT,      // 事件帧，FIFO语义
    CAN_MSG_SDO,        // 可靠事务帧，ACK+重发
    CAN_MSG_HEARTBEAT,  // 心跳帧
} CanMsgType;

typedef struct {
    CanBusId   bus_id;         /* ★ 双总线切换关键字段 */
    uint32_t   id;
    uint8_t    dlc;
    uint8_t    data[8];
    CanMsgType type;
    CanPrio    priority;
    uint32_t   enqueue_tick;   // 队列积压检测
} CanTxFrame;

2.2 PDO Buffer（实时状态槽）

PDO 不使用普通队列，而是每个控制节点持有一个覆写槽：

typedef struct {
    CanTxFrame latest;    // 永远只保存最新值
    uint8_t    updated;   // dirty flag
    uint32_t   node_id;
} PdoBuffer;

// 控制器更新（不阻塞）
static inline void pdo_update(PdoBuffer *b, const CanTxFrame *f) {
    b->latest  = *f;
    b->updated = 1;     // 原子写，旧帧隐式丢弃
}

// 调度器取帧（由 Budget 驱动）
bool pdo_pop(PdoBuffer *b, CanTxFrame *out) {
    if (!b->updated) return false;
    *out = b->latest;
    b->updated = 0;
    return true;
}
v1.0 / v2.0 问题 原文档将 PDO（实时控制帧）与 FIFO 混用，导致旧控制值积压。PDO Buffer 与 FIFO 是并列层级，不是替代关系：PDO Buffer 在上层处理覆写，FIFO 在下层处理事务消息保序。

发送语义：PDO Buffer vs FIFO

维度PDO BufferEVENT / SDO FIFO
数据关系覆写（新替旧）累加（严格保序）
是否保留历史否是
队列满行为无队列，直接覆写丢最旧 + 计数
适用消息位置/力矩/速度指令夹爪事件 / SDO配置
调度驱动PDO Budget 周期取帧空闲时 FIFO 出队
与FSM关系ZeroErr PP 事务由FSM管理，不走PDO层SDO层不重叠已有FSM重试逻辑
Priority Starvation 警告 若 PDO 无限优先（只要有 PDO 就跳过其他），Heartbeat / SDO / Event 永远发不出去。必须引入 Bus Budget 机制约束 PDO 的带宽占用比例。

Bus Budget：总线预算机制

1Mbps CAN 总线，一帧约 130 bit，1ms 内理论最大约 7 帧。通过预算分配防止优先级饿死：

PDO ×4
EVT ×1
SDO ×1
HB ×1
PDO 实时控制 4/7 EVENT 事件 1/7 SDO 配置 1/7 Heartbeat 1/7 
typedef struct {
    uint8_t pdo_budget;    // 每ms可发PDO帧数
    uint8_t event_budget;
    uint8_t sdo_budget;
    uint8_t hb_budget;
} BusBudget;

// 1ms 定时器中断刷新预算
void bus_budget_refresh(BusBudget *b) {
    b->pdo_budget   = 4;
    b->event_budget = 1;
    b->sdo_budget   = 1;
    b->hb_budget    = 1;
    // 高负载时可动态缩减 PDO budget
    if (bus_load_get() > 80) b->pdo_budget = 2;
}

调度器实现（修正版）

5.1 Phase Shift 的根本问题

v3.0 的 Phase Shift 实现存在精度不匹配缺陷：

设计缺陷 Phase 单位是 μs，但调度器是 1ms 定时器 tick。同一 tick 内所有到期的 job 一次性入队，200μs 的 phase 偏移完全无效——所有电机仍然同时入队。
方案机制代价适用场景
方案A：μs 级定时器定时器改为 100μs tick，phase 单位对齐中断频率提高 10×，CPU 开销增加需要严格帧级时序对齐
方案B：next_tick 自然错峰每个电机自己的 next_tick 驱动，Budget 做上限保护实现最简，无额外开销✓ 推荐：多电机闭环控制
方案C：TDMA 时间片预分配发送窗口，确定性调度静态资源占用，电机数量受限电机数量固定、需要硬实时

5.2 正确的调度器：next_tick 驱动，Budget 保护

驱动力是每个电机自己的 next_tick，Budget 只是上限保护，不是"凑满配额去找电机发"。

// 定时器中断 1ms tick
void CAN_Scheduler_Tick(CanBusId bus)
{
    uint32_t  now = get_tick_us();
    BusBudget *b  = &g_budget[bus];

    for (uint8_t i = 0; i < s_job_count[bus]; i++) {
        CanJob *job = &s_jobs[bus][i];

        if (now < job->next_tick) continue;  // 未到期，跳过

        if (b->pdo_budget == 0) continue;    // budget 耗尽，推到下一tick
        // next_tick 不更新：下次tick继续尝试，不丢失这次发送机会

        CanTxFrame frame = {.bus_id = bus, .type = CAN_MSG_PDO};
        if (pdo_pop(&s_pdo_buf[i], &frame)) {   // 有新值才发
            can_tx_enqueue(&frame);
            b->pdo_budget--;
        }
        // 无论是否发送，next_tick 都推进——电机周期不受控制器写入频率影响
        job->next_tick += job->period_us;
    }

    // EVENT / SDO / HB 按各自 budget 处理（同 v3.0）
}
关键设计点 到期了但 budget 耗尽 → next_tick 不更新，下一 tick 重试。到期了 PDO 槽没有新值 → next_tick 照常推进，不阻塞周期。两者行为分开处理。

RX 反馈：全双工负载建模

v3.0 的 Bus Load 统计只计了 TX 侧。CAN 总线是共享介质，RX 帧同样占用带宽，必须双向建模。

7.1 三种反馈模式

模式触发方式流量特征危险性
SDO 响应发一条指令，延迟后返回单帧低频、不定时低
异步 TPDO节点状态变化自主上报随机、不可预测中
同步 TPDO 风暴发 SYNC 帧，所有节点同时回瞬时大量，N×M 帧极高，主要拥塞源

7.2 TPDO 风暴的量化

最坏情况计算 6个电机 × 3个TPDO/电机 = 18帧同时竞争总线。130μs × 18 = 2340μs，超过两个 1ms 控制周期。下一轮 PDO 指令发送被整体阻塞。 
// 全双工 Bus Load 正确公式
// Bus_Load = (TX_bits + RX_bits) / (window_us × bitrate_Mbps)

void HAL_CAN_RxFifo0MsgPendingCallback(CAN_HandleTypeDef *hcan) {
    CAN_RxHeaderTypeDef hdr;
    uint8_t data[8];
    HAL_CAN_GetRxMessage(hcan, CAN_RX_FIFO0, &hdr, data);

    CanBusId bus = (hcan == &hcan1) ? CAN_BUS_1 : CAN_BUS_2;

    // 纳入负载统计（与TX共享同一统计器）
    bus_stats_on_rx(&g_stats[bus], hdr.DLC);

    // 总线录播（见第12节）
    can_log(bus, DIR_RX, hdr.StdId, hdr.DLC, data);

    // 分发到对应电机驱动
    can_rx_dispatch(bus, hdr.StdId, hdr.DLC, data);
}

7.3 TPDO 风暴抑制：Inhibit Time

通过 SDO 配置每个节点的 Inhibit Time，强制同一 PDO 两次上报之间的最小间隔，把风暴展开成流：

// 按节点编号错开 Inhibit offset，单位 100μs
for (uint8_t i = 0; i < motor_count; i++) {
    uint16_t inhibit_100us = i * 2;  // 每个电机错开 200μs
    sdo_write(motors[i].node_id,
              0x1800, 0x03,          // TPDO1 Inhibit Time 对象
              inhibit_100us);
}
// 效果：Motor0 t=0μs，Motor1 t=200μs，Motor2 t=400μs...

7.4 TPDO 频率分级

数据类型建议频率理由
位置 + 速度跟随 SYNC（500Hz）闭环必须，不可降频
电流100Hz（每5个SYNC发一次）监控用，不需要高频

7.5 发送预算纳入 RX 预留

void bus_budget_refresh(BusBudget *b, uint8_t active_motors) {
    uint8_t max_frames = 7;                       // 1ms @ 1Mbps 理论最大帧数
    uint8_t rx_reserved = active_motors * 1;     // 每电机预留1帧TPDO（位置+速度）
    uint8_t tx_total    = max_frames - rx_reserved;

    b->pdo_budget   = tx_total * 4 / 7;          // TX 中 PDO 占 4/7
    b->event_budget = 1;
    b->sdo_budget   = 1;
    b->hb_budget    = 1;

    if (g_stats->load_pct > 80) b->pdo_budget >>= 1; // 过载动态降级
}

TDMA 时间片调度

8.1 TDMA 本质

Time Division Multiple Access（时分多址）：将总线周期切成 N 个时间片，每片预分配给一个电机。到了它的时间片，它可以发也可以不发；其他时间片它不发。

维度Budget 限流（v3.0）TDMA 时间片（v4.0）
调度方式统计限流，到期竞争预分配，按槽发送
最坏延迟不确定（budget 耗尽时推迟）严格确定（最多等一个周期）
总线利用率高（空闲 budget 可被占用）低（时间片空着就浪费）
实现复杂度低中
工业标准CANopen PDO + InhibitTTCAN（ISO 11898-4）

8.2 时间片结构

1ms 周期，6个电机，每片 ~150μs：

│◀──────────────────────── 1000μs ────────────────────────▶│
│  M0  │  M1  │  M2  │  M3  │  M4  │  M5  │  EVENT/SDO  │
│150μs │150μs │150μs │150μs │150μs │150μs │    100μs    │

每片内：
├─ TX: 该电机 PDO 指令（~130μs）
└─ RX: 该电机 TPDO 反馈（同窗口内自然返回）
TX 与 RX 对齐同一时间片，不互相干扰。

8.3 TDMA 与 Budget 的矛盾：电机数量超限

静态分配的问题 12个电机静态分配 → 1ms ÷ 12 = 83μs/片。一帧 130μs，一个时间片放不下一帧。TDMA 对电机数量有硬上限：最大电机数 = 1000μs ÷ 每片最小宽度。

8.4 解决方案：动态活跃集合

不给所有电机预留时间片，只为当前活跃的电机分配：

typedef enum {
    MOTOR_SLEEP,    // 不占时间片，只发心跳
    MOTOR_HOLD,     // 低频时间片（50Hz），保持位置
    MOTOR_ACTIVE,   // 高频时间片（500Hz），闭环运动
} MotorSchedState;

// 活跃集合变化时重新分配时间片
void tdma_recompute(CanBusId bus) {
    uint8_t active_count = 0;
    for (uint8_t i = 0; i < motor_count; i++)
        if (motors[i].sched == MOTOR_ACTIVE) active_count++;

    uint32_t slot_us = (1000 - 100) / active_count; // 预留100μs给EVENT
    uint8_t  slot_idx = 0;
    for (uint8_t i = 0; i < motor_count; i++) {
        if (motors[i].sched != MOTOR_ACTIVE) continue;
        s_tdma_slots[bus][slot_idx].motor_idx  = i;
        s_tdma_slots[bus][slot_idx].start_us   = slot_idx * slot_us;
        slot_idx++;
    }
    s_tdma_slot_count[bus] = active_count;
}

8.5 三级调度状态

状态发送频率时间片典型场景
MOTOR_ACTIVE500Hz（2ms）占有高频时间片摆动腿、正在运动的关节
MOTOR_HOLD50Hz（20ms）低频竞争发送支撑腿、保持位置的关节
MOTOR_SLEEP1Hz 心跳不占控制时间片待机电机、暂停关节
工程建议 TDMA 适合电机数量固定、需要硬实时保证的场景。如果同时活跃电机数量变化频繁（秒级以下切换），重计算开销本身会引入抖动，此时 Budget 限流反而更稳定。两者可以混用：ACTIVE 电机用 TDMA，HOLD/SLEEP 电机用 Budget 竞争。

总线负载可观测

CAN 控制器不上报带宽占用。Bus Load 的正确定义是总线 bit 占用时间 / 总时间，而非帧数比例。TX 和 RX 必须共同纳入统计。

9.1 统计实现

typedef struct {
    uint32_t used_bits;      // 当前窗口 TX+RX 总 bit 数
    uint8_t  load_pct;      // 0~100%
    uint32_t tx_drop_count; // TX 丢帧计数
    uint32_t rx_frame_count;// RX 帧总数
    uint16_t queue_depth;   // 当前队列水位
} BusStats;

// TX / RX 中断各调用一次
void bus_stats_on_frame(BusStats *s, uint8_t dlc) {
    // 标准帧：SOF(1)+仲裁(11)+控制(6)+数据(dlc×8)+CRC(16)+ACK(2)+EOF(7) + 填充估算
    s->used_bits += 44 + dlc * 8 + 25;
}

// 10ms 窗口计算，在定时器中断里调用
void bus_stats_window_update(BusStats *s) {
    s->load_pct  = (uint8_t)(s->used_bits / 100); // 10000μs × 1bit/μs = 10000
    s->used_bits = 0;
}

9.2 动态降级触发

if (g_stats[bus].load_pct > 80) {
    g_budget[bus].pdo_budget >>= 1;          // PDO 减半

}

抗拥塞设计

总线拥塞不只是设计问题，也是运行时问题。典型触发场景：节点同时上电、SDO 重发风暴、心跳相位重合、SYNC 触发 TPDO 风暴。

手段机制解决的问题
next_tick 自然错峰注册时设置不同初始 next_tick多电机固定周期帧自然分散
Bus Budget每 ms 限制各类型帧数上限防止 PDO 饿死其他类型
Inhibit TimeSDO 配置节点 TPDO 最小发送间隔TPDO 风暴展开成流
TPDO 频率分级位置高频，电流/温度低频降低 RX 总带宽占用
动态降级负载 >80% 压缩 PDO budget突发流量 / 异常节点重发
PDO 覆写槽旧控制帧被新值隐式丢弃防止控制帧在队列积压
SDO 发送互斥等待响应期间 sdo_budget=0防止 SDO 请求与响应交错

SDO 响应窗口保护

// SDO 状态机：发出请求后冻结 sdo_budget，直到收到响应

CiA402 分布式电机架构

分布式控制的本质：主控发目标值，节点本地闭环。主控不控制 PWM，只发位置 / 力矩 / 速度目标。

步态规划 / 轨迹生成
Command Aggregator

可观测性：总线录播器

现有系统人肉串行发送，大部分时候稳定，偶发不稳定但找不到原因。根本原因是总线上发生了什么不可见。最小成本的解决方案：环形 buffer 录播器，出问题时回放时序。

12.1 录播器实现

typedef struct {
    uint32_t tick_us;   // 微秒时间戳
    uint8_t  bus;       // CAN_BUS_1 / CAN_BUS_2
    uint8_t  dir;       // DIR_TX / DIR_RX
    uint32_t id;
    uint8_t  dlc;
    uint8_t  data[8];
    uint8_t  dropped;   // 此前是否有丢帧
} CanLogEntry;

#define CAN_LOG_SIZE  256
static CanLogEntry g_can_log[CAN_LOG_SIZE];
static uint8_t     g_log_idx = 0;

#define CAN_LOG(bus, dir, id, dlc, data_ptr)                \
    do {                                                        \
        CanLogEntry *e = &g_can_log[g_log_idx++ % CAN_LOG_SIZE]; \
        e->tick_us = get_tick_us();                            \
        e->bus = bus; e->dir = dir; e->id = id; e->dlc = dlc;  \
        memcpy(e->data, data_ptr, dlc);                          \
        e->dropped = g_stats[bus].tx_drop_count > 0;           \
    } while(0)

12.2 在 TX 和 RX 各插一行

// TX 侧（邮箱中断搬运时）
HAL_CAN_AddTxMessage(hcan, &hdr, frame.data, &mb);
CAN_LOG(bus, DIR_TX, frame.id, frame.dlc, frame.data);

// RX 侧（接收中断）
HAL_CAN_GetRxMessage(hcan, CAN_RX_FIFO0, &hdr, data);
CAN_LOG(bus, DIR_RX, hdr.StdId, hdr.DLC, data);

12.3 出问题时的分析

通过调试器查看 g_can_log 环形 buffer，可以直接看到：

观察到的现象对应的问题
TX 帧和 RX 帧 tick_us 高度重叠RX 风暴阻塞了 TX 发送窗口
某电机连续多帧无 RX 回复节点掉线 / 总线错误 / ID 配置错误
dropped=1 出现队列在该时刻满过，有帧被丢弃
TX 帧间隔异常拉长邮箱被 RX 处理长时间占用 / 中断优先级问题

成本收益 256 条 log，每条约 20 字节，总计 5KB RAM。覆盖最近 256 帧事件，足够定位绝大多数偶发问题。这是整个架构里成本最低、收益最高的单项投资。

邮箱中断 while 的设计意图

邮箱空中断回调里使用 while 而非 if，原因是一次中断可能对应多个空闲邮箱。

13.1 while 不会阻塞的原因

循环有两个退出条件，任意一个满足立即退出，没有任何等待：

while (HAL_CAN_GetTxMailboxesFreeLevel(hcan) > 0) {
    CanTxFrame frame;
    if (!tx_queue_pop(...)) break;   // 退出条件1：队列空，立即 break
    HAL_CAN_AddTxMessage(...);         // 寄存器写入，微秒级，不等待
}                                      // 退出条件2：FreeLevel==0，条件为假退出
有界循环 硬件邮箱只有 3 个，FreeLevel 最大返回 3。while 最多执行 3 次，执行时间约几微秒，在 168MHz STM32F405 上完全可以接受。

13.2 为什么不用 if

写法行为问题
if (FreeLevel > 0)每次中断只填 1 帧3 个空邮箱需要 3 次中断才能填满，吞吐量低
while (FreeLevel > 0)一次中断填满所有空邮箱吞吐量最大化，队列积压快速消化

13.3 需要注意的地方

FromISR API 如果 tx_queue_pop 内部使用了 FreeRTOS 临界区保护，在中断上下文里必须使用 taskENTER_CRITICAL_FROM_ISR() 而非 taskENTER_CRITICAL()，否则会造成调度器断言失败。

定时器中断的作用

TIM_PeriodElapsedCallback 是 STM32 通用定时器的溢出中断回调。定时器计数器从 0 计数到 ARR 设定值后溢出，触发一次中断，然后自动复位重新计数。

计数器：0 → 1 → 2 → ... → ARR → 溢出 → 中断 → 复位为0 → 继续
                                          ↓
                             TIM_PeriodElapsedCallback()
                                          ↓
                             bus_budget_refresh()   重置本ms配额
                                          ↓
                             CAN_Scheduler_Tick()   检查所有 next_tick

14.1 为什么不用 HAL_GetTick() 轮询

对比项主循环 HAL_GetTick() 轮询定时器中断
触发精度取决于主循环负载，可能 ±几ms硬件保证，μs 级精度
500Hz 控制周期抖动负载重时抖动明显可忽略
与业务代码耦合强耦合，主循环卡住调度停止完全解耦
实现复杂度简单需配置定时器

TPDO 对发送的影响

TPDO 反馈帧不会导致主控发送失败，但会造成发送延迟。理解这个区别对系统调优非常重要。

15.1 CAN 仲裁机制保证不丢失

主控发送 PDO 与电机 TPDO 同时竞争总线时，CAN 硬件按 ID 数值大小自动仲裁，ID 小的先发，ID 大的自动退让并等待重试：

主控 PDO   ID=0x200  ──┐
                        ├─ 仲裁 → ID小的先发，另一个硬件自动重试
电机 TPDO  ID=0x180  ──┘         两帧都不丢失
退让帧硬件自动重发 仲裁失败的帧由 CAN 控制器硬件自动重发，对软件完全透明，不需要任何处理。

15.2 TPDO 造成的发送延迟

主控想在 t=0 发 PDO
t=0:    TPDO 正在总线上，主控 PDO 退让等待
t=130:  TPDO 发完，主控 PDO 开始发送
→ 延迟 130μs（一帧时间）

TPDO 风暴时的最坏情况：

发出 SYNC 帧
↓
6个电机 × 3个TPDO = 18帧同时竞争总线
↓
主控下一条 PDO：等待 18 × 130μs = 2340μs ≈ 2.3ms
↓
控制周期 2ms → 整整一个周期的指令被延迟

15.3 TDMA 时间片与 TPDO 的关系

澄清：时间片无法控制 RX 回来的时机 TDMA 时间片只管 TX 发送时刻，TPDO 回复时机由电机节点固件决定，主控无法控制。所谓"TX 与 RX 对齐同一时间片"是不准确的说法——实际上 TPDO 可能在任意时刻回来，由 CAN 硬件仲裁自动处理。
场景结果
主控 TX 与单个 TPDO 竞争ID 小的先发，另一个等待重发，无损失，延迟约 130μs
TPDO 风暴（多帧同时竞争）TX 被延迟最多几毫秒，不丢失，影响控制周期
NART 模式开启仲裁失败直接丢帧，不要对控制帧开启
总线错误（Bus-Off）所有发送停止，需要恢复机制

自动重传风险与故障隔离

16.1 自动重传拖垮控制器的场景

风险链条 电机掉线 / 总线断路 → 无节点回复 ACK → 发送失败 → 自动重传 → 再失败 → 再重传 → 邮箱永久占用 → 后续帧全部阻塞 → TEC 溢出 → Bus-Off → 整条总线失联

16.2 Bus-Off 触发机制

TEC（发送错误计数）：每次发送失败 +8，每次成功 -1
REC（接收错误计数）：类似机制

TEC > 127：Error Passive（被动错误，影响减小）
TEC > 255：Bus-Off，控制器完全停止收发

16.3 CAN 总线能检测什么，不能检测什么

重要认知 CAN 硬件检测的是总线电气层错误，无法区分"目标节点掉线"和"发送成功"——因为其他节点会代替掉线节点回复 ACK，硬件层面发送永远"成功"。单节点掉线只能由应用层软件检测。
故障类型谁检测检测方式
总线断路 / 短路CAN 硬件ACK 错误 → TEC 增加 → Bus-Off
节点持续发错误帧CAN 硬件该节点 TEC 溢出 → 自动 Bus-Off 隔离，不拖垮其他节点
单节点静默掉线应用层软件RX 超时检测 / Heartbeat 丢失
节点回复数据异常应用层软件数据校验 / 状态值范围检查

16.4 推荐配置策略

// PDO 控制帧：关闭自动重传（NART）
// 丢一帧没关系，下帧覆写槽会覆盖，与 PDO 语义完全一致
hcan.Init.AutoRetransmission = DISABLE;

// Bus-Off 硬件自动恢复
// 等待 128×11 个隐性位（约1.4ms @1Mbps）后自动重新加入总线
hcan.Init.AutoBusOff = ENABLE;

16.5 邮箱超时看门狗

即使开启 NART，也应加软件看门狗作为兜底，防止极端情况下邮箱卡死：

void CAN_TxWatchdog(CanBusId bus) {
    uint32_t now = get_tick_ms();
    for (uint8_t mb = 0; mb < 3; mb++) {
        if (!mailbox_pending[bus][mb]) continue;
        if (now - mailbox_send_tick[bus][mb] > TX_TIMEOUT_MS) {
            HAL_CAN_AbortTxRequest(s_can_hdl[bus], 1 << mb);
            g_stats[bus].tx_timeout_count++;
            log_warn("CAN%d mailbox%d timeout, aborted", bus, mb);
        }
    }
}
// 在 1ms 定时器中断或主循环里周期调用

16.6 单节点掉线软件检测

// 方法一：RX 超时（适合有 TPDO 反馈的电机）
void motor_health_check(MotorDevice *m) {
    if (get_tick_ms() - m->last_rx_tick > RX_TIMEOUT_MS) {
        m->state = MOTOR_FAULT_OFFLINE;
        log_warn("Motor 0x%03X offline", m->node_id);
    }
}

// 方法二：Heartbeat 监控（CANopen 标准方式）
// 电机每 100ms 发一帧心跳，主控监控间隔
if (get_tick_ms() - m->last_heartbeat_tick > 200) {
    m->state = MOTOR_FAULT_OFFLINE;
}

16.7 分层故障隔离

单节点掉线（静默）
    ↓
应用层：RX 超时 / Heartbeat 丢失 → 标记 MOTOR_FAULT_OFFLINE
    ↓
步态层：该电机 FAULT → 切换步态 / 降级运行

节点持续发错误帧
    ↓
CAN 硬件：该节点 TEC 溢出 → 自动 Bus-Off 隔离
    ↓
其他节点和主控不受影响，继续正常通信

整条总线故障
    ↓
Bus-Off 触发 → AutoBusOff 硬件恢复
    ↓
软件：检测 Bus-Off 事件 → 记录日志 → 重新初始化 → 上报系统故障
    ↓
每一层只处理自己能处理的，超出范围往上抛

工程原则总结

PRINCIPLE · 01
中断只搬运
邮箱空中断不决定发什么，只从队列取帧填充邮箱。调度策略全部在 Scheduler 里。
PRINCIPLE · 02
应用层不碰 HAL
业务代码只调用 pdo_update() 或 can_tx_enqueue()，不直接调用 HAL_CAN_AddTxMessage。
PRINCIPLE · 03
帧携带 bus_id
消息帧自描述走哪条总线。电机只持有 bus_id，切换总线只改一个字段，发送路径零修改。
PRINCIPLE · 04
控制帧不排队
PDO 使用覆写槽，旧控制值被新值隐式丢弃，不在队列积压产生控制滞后。
PRINCIPLE · 05
全双工负载建模
TX 和 RX 共享同一 bit 统计器。只统计 TX 会低估真实负载，TPDO 风暴是主要 RX 拥塞源。
PRINCIPLE · 06
非实时消息必须可降级
Bus Budget 在高负载时压缩 PDO 配额，为 Heartbeat / SDO 保留带宽，防止饿死。
PRINCIPLE · 07
next_tick 驱动，Budget 保护
调度驱动力是每个电机自己的 next_tick，Budget 只是上限保护，不是"凑满配额去找电机发"。
PRINCIPLE · 08
RX 反馈需要治理
Inhibit Time 错开 TPDO 上报时刻；位置高频，电流低频；按需 SDO 查询诊断数据，不主动上报。
PRINCIPLE · 09
不重复可靠层
已有 FSM 管理事务确认的设备（ZeroErr PP）不再叠加 SDO 重发层，避免双重重试冲突。
PRINCIPLE · 10
先观测再优化
录播器是成本最低收益最高的投资。偶发问题先看日志，确认根因后再加机制，不盲目复杂化。
PRINCIPLE · 11
TDMA 按需引入
电机数量固定且需要硬实时时用 TDMA；电机数量动态变化时 Budget 限流更稳定。两者可混用。
PRINCIPLE · 12
PDO 关闭自动重传
控制帧开启 NART：丢一帧下帧覆盖，与覆写槽语义一致。防止单节点故障导致邮箱卡死拖垮整条总线。
PRINCIPLE · 13
单节点掉线靠软件检测
CAN 硬件无法检测单节点静默掉线——其他节点会代为回复 ACK。必须用 RX 超时或 Heartbeat 丢失在应用层检测。
PRINCIPLE · 14
分层故障隔离
单节点故障由应用层处理，不传播到总线；总线故障由 AutoBusOff 恢复，不传播到系统崩溃。每层只处理自己能处理的。
PRINCIPLE · 15
扩展不改接口
增加 CAN3 只需扩展枚举和句柄数组；增加电机协议只需新增驱动文件；应用层只操作电机对象。
